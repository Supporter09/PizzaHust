\section{Analysis}

\subsection{General Use Case Overview}

This section presents the use case analysis for the \systemname\ Online Ordering System. The scope is limited to a web-based MVP that can be delivered within 3.5 months while still satisfying the mandatory business requirements of the project topic. The refined MVP supports menu browsing, pizza customization, combo promotions, cash on delivery, guest order-code tracking, customer account features, loyalty points, kitchen order queue handling, third-party delivery integration, and basic reporting for the store owner.

\subsubsection{General Use Case Diagram}

\begin{figure}[H]
    \centering
    \includegraphics[width=1\linewidth]{../PizzaHust.png}
    \caption{\systemname\ General Use Case Diagram}
    \label{fig:pizzahust-general-use-case}
\end{figure}

The figure above provides the high-level use case overview for \systemname. The detailed descriptions below follow the final MVP boundary used in this report. In particular, AI recommendation is treated as a future enhancement rather than an MVP use case, and the kitchen workflow is refined into a \textbf{Kitchen Order Queue} that helps staff process the most urgent orders first.

The final MVP includes four primary internal actors and one external supporting actor:

\begin{description}
    \item[Guest:] A visitor who can browse menus, customize pizzas, place a cash-on-delivery order, and track an order by using a generated order code.
    \item[Customer:] A registered user who inherits all guest capabilities and can additionally log in, manage profile information, view order history, earn loyalty points, and redeem points during checkout.
    \item[Admin / Store Owner:] The business operator who manages pizzas, toppings, side dishes, categories, combo campaigns, customer records, order supervision, and sales reports.
    \item[Kitchen Staff:] Internal staff members who work from a prioritized kitchen queue, process the next recommended order, update preparation progress, and mark completed orders as ready for dispatch.
    \item[Third-Party Delivery Service:] An external delivery platform integrated through API to receive delivery requests and return delivery status updates.
\end{description}

\subsubsection{Use Case Relationships}

The main relationships among use cases are summarized below:

\begin{itemize}
    \item \textbf{Customer} is modeled as a specialized actor that inherits all use cases available to \textbf{Guest}.
    \item \textbf{Customize Pizza} includes selecting pizza size and crust type, and it optionally extends to adding extra toppings.
    \item \textbf{Place COD Order} includes providing delivery information, calculating the fixed delivery fee, and confirming cash on delivery.
    \item \textbf{Redeem Points for Discount} extends \textbf{Place COD Order} and is available only to authenticated customers with sufficient loyalty points.
    \item \textbf{Track Order} includes \textbf{Synchronize Delivery Status} when the order has already been sent to the third-party delivery service.
    \item \textbf{Mark Order Ready for Dispatch} includes \textbf{Request Delivery Service} when the order is packed and ready for handoff.
    \item \textbf{Process Prioritized Order} depends on the prioritized kitchen queue that ranks waiting orders by urgency, promised time, and readiness constraints.
\end{itemize}
